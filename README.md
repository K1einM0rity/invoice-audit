# 智能财务审计助手

基于多模态视觉 API + 自研规则引擎的发票自动审计系统。上传发票图片，自动识别11个字段，**11条规则校验**，生成审计建议，导出Excel。

全程零训练，Python + Streamlit，CPU 即可运行。

## 功能

- **批量上传** — 支持并发处理（4线程），一次可上传多张 JPG/PNG
- **视觉识别** — 多模态模型直接看图提取字段，带置信度评分
- **QR码交叉验证** — 自动解码发票二维码，与AI识别结果比对
- **11条规则校验** — 号码格式、税号两层防御、金额验算、日期逻辑、必填项、重复检测、发票代码、类型交叉验证、税率一致性、字段置信度、QR码比对
- **智能审计建议** — 基于校验结果自动生成审计意见
- **统计看板** — 合规率、异常数、警告数、金额合计
- **Excel导出** — 双Sheet（审计结果 + 异常明细），自动列宽调整
- **跨批次查重** — 发票号码持久化存储，自动检测历史重复

## 技术架构

```
用户上传发票图片
        ↓
  vision_extractor.py    ← 多模态视觉API，提取字段+置信度
        ↓
  qr_decoder.py          ← QR码解码，提取结构化数据
        ↓
  validator.py           ← 自研规则引擎，11条规则链
        ↓
  auditor.py             ← LLM审计建议生成
        ↓
  pipeline.py            ← 流水线编排，异常兜底
        ↓
  app.py (Streamlit)     ← Web界面 + 统计看板 + Excel导出
```

## 校验规则详解

| 规则编号 | 规则名称 | 严重性 | 说明 |
|---------|---------|--------|------|
| FR-D01 | 发票号码格式 | ERROR | 8位（传统发票）或20位（数电发票）纯数字 |
| FR-D02 | 统一社会信用代码 | ERROR/WARNING | 18位格式校验（ERROR）+ ISO 7064 MOD 31-3校验码验证（ERROR） |
| FR-D03 | 开票日期逻辑 | ERROR/WARNING | 日期格式解析（ERROR）、晚于当前（ERROR）、超过180天（WARNING） |
| FR-D04 | 金额验算 | ERROR/WARNING | Decimal精确计算：金额+税额≈价税合计（容差0.01元），缺失时跳过（WARNING） |
| FR-D05 | 必填项缺失 | ERROR | 发票号码、开票日期、购买方名称、金额为核心必填字段 |
| FR-D06 | 重复报销检测 | ERROR/WARNING | 同批次重复（ERROR）、历史记录重复（ERROR）、缺字段无法检测（WARNING） |
| FR-D07 | 发票代码格式 | ERROR | 10位纯数字（传统发票）或12位且首位为0（电子发票），数电发票无代码跳过 |
| FR-D08 | 发票类型交叉验证 | WARNING | 号码位数与类型一致性检查、代码号码互换检测 |
| FR-D09 | 税率一致性检查 | WARNING | 税额÷金额 vs 标准税率（13%/9%/6%/5%/3%/1%/0%），偏差>0.5%告警 |
| FR-D10 | 字段置信度检查 | WARNING | 关键字段OCR置信度低于阈值（2-3分/5分）时提示人工复核 |
| FR-D11 | QR码交叉验证 | ERROR | QR码解码数据与AI识别结果比对，不一致时报错（以QR码为准） |

### 严重性分级说明

- **🔴 ERROR（红色）**：导致发票无法入账或认证的严重问题，必须阻断流程
  - 上传图片识别为非发票
  - 发票号码、金额、购买方名称等核心必填字段缺失
  - 统一社会信用代码格式错误或校验码不匹配
  - 金额验算失败、日期格式错误、重复报销
  - QR码与AI识别结果不一致

- **🟡 WARNING（黄色）**：需人工复核但不影响入账的问题
  - 日期超过180天（企业报销期限可配置）
  - 税率轻微偏差（可能为OCR识别错误）
  - 非企业抬头（个人抬头发票）
  - 字段置信度较低、发票类型缺失

- **🟢 PASS（绿色）**：校验通过，无明显异常

## 技术栈

- **视觉识别**：Agnes-2.0-Flash 多模态大模型
- **Web框架**：Streamlit
- **编程语言**：Python 3.11
- **数据处理**：pandas, openpyxl, Pillow, Decimal
- **图像处理**：OpenCV（QR码解码）
- **网络请求**：requests
- **并发处理**：ThreadPoolExecutor（4线程并发）

## 性能特点

- **并发处理**：支持4线程同时处理多张图片，提升批量审计效率
- **异常兜底**：每个环节都有异常处理，单张图片失败不影响整体流程
- **智能重试**：API超时、网络异常等情况有友好提示
- **内存优化**：图片在主线程预读，避免子线程竞争
- **持久化存储**：历史发票号码JSON存储，跨会话查重

## 合规声明

- 本项目仅做发票表面信息校验（格式、逻辑、一致性）
- **不做真伪查验**（需对接税务局官方接口：https://inv-veri.chinatax.gov.cn）
- 校验规则依据：
  - GB/T 32100-2015《法人和其他组织统一社会信用代码编码规则》（现行有效）
  - 国家税务总局公告（截止2026年6月）
  - ISO 7064 MOD 31-3 校验码算法
- 企业内部报销期限（180天）为可配置默认值（见 `validator.py` 中 `DEADLINE_DAY`），非税法强制
- 测试必须使用模拟发票或网络公开样本，严禁使用真实敏感票据

## 常见问题

**Q: 为什么有些发票显示黄色警告而不是红色错误？**  
A: 黄色表示需要人工复核但不至于拒收的问题（如日期超期、税率偏差）。红色表示必须处理的严重问题（如号码缺失、重复报销）。详见"严重性分级说明"。

**Q: 如何修改报销期限天数？**  
A: 编辑 `validator.py` 文件第11行的 `DEADLINE_DAY = 180`，改为所需天数即可。

**Q: 历史查重数据存在哪里？**  
A: 项目根目录的 `invoice_history.json` 文件，会自动保存已处理的发票号码。

**Q: 支持哪些发票类型？**  
A: 增值税专用发票、增值税普通发票、增值税电子普通发票、增值税电子专用发票、数电发票（专票/普票）等主流类型。
